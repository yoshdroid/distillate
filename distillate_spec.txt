Distillate 仕様書（original/distillate.py 解析ベース）

1. 文書の目的

この文書は、未完成の `original/distillate.py` を解析し、今後 AI で再実装しやすいように仕様を整理したものである。
元コードは Pyxel を用いたグリッド型の簡易水流シミュレーションであり、プレイヤーは一時的なブロックを描いて水の流れを誘導する。

この仕様書では以下を区別して扱う。

- 現行コードから読み取れる「実際の挙動」
- 現行コードに含まれる未完成点・不具合・曖昧さ
- 再実装時に採用すべき「AI 実装しやすい仕様」


2. ゲーム概要

- 2D グリッド上で水が湧き出し、重力に従って落下し、障害物に当たると左右へ流れる。
- プレイヤーはマウスドラッグで一時ブロックを配置し、水の経路を制御する。
- ステージ内には水源タイルと排水タイルが存在する。
- 排水タイルの隣接マスに存在する水は除去される。
- 勝敗条件やスコアは元コードには未実装であり、現時点では「水の流れを眺めて操作するサンドボックス」に近い。


3. 実行環境と画面仕様

- ライブラリ: Pyxel
- 1 マスのサイズ: 8 px
- 画面サイズ:
  - WIDTH = 249 px
  - HEIGHT = 185 px
- 論理グリッドサイズ:
  - X 方向 31 マス
  - Y 方向 23 マス
- グリッド座標は 0 始まりとする
- 描画上はグリッド線が表示される

補足:
元コードでは `WIDTH = 256 - 7`, `HEIGHT = 192 - 7` としており、31x23 マスを 8px 単位で収めるための調整になっている。


4. ステージデータ仕様

4.1 タイル種別

- 0: 空きマス
- 1: 固定壁
- 2: 水源
- 3: 排水口

4.2 元コードに埋め込まれている初期ステージ

- サイズは 31 列 x 23 行
- 水源は 1 個
  - (15, 3)
- 排水口は 2 個
  - (15, 16)
  - (3, 19)

4.3 タイルの衝突扱い

- 固定壁 `1` は通行不可
- 水源 `2` は水生成地点であり、空間としては占有タイル扱い
- 排水口 `3` も空間としては占有タイル扱い
- 元コードでは「0 以外のタイルはすべて障害物」として扱われている


5. エンティティ仕様

5.1 Water

役割:
- 水 1 マス分の粒子

保持情報:
- `x`, `y`: 現在のグリッド座標
- `vect`: 現在の移動ベクトル。候補は `[-1,0]`, `[1,0]`, `[0,1]`, `[0,0]`
- `status`: 元コードでは未活用。描画色切替用の名残
- `stress`: 元コードでは未活用
- `birth`: 生成フレーム。元コードではデバッグ用途のみ

基本挙動:
- 下マスが空いていれば下へ 1 マス移動
- 下マスが塞がっていれば左右どちらかへ流れようとする
- 左右の向きは初回にランダム選択される
- 選んだ側が塞がっていれば停止する
- 元コードでは「反対側へ再挑戦する」ロジックがコメントアウトされているため、横移動の挙動は未完成

5.2 Block

役割:
- プレイヤーが配置する一時的な壁

保持情報:
- `x`, `y`: グリッド座標
- `life`: 残り寿命
- `anime`: 劣化アニメ段階
- `birth`: 生成フレーム

基本挙動:
- 空きマスにのみ配置可能
- 寿命は初期値 200 フレーム
- 毎フレーム 1 ずつ減少
- 残量に応じて点滅頻度が増す
- 寿命 0 で消滅


6. 入力仕様

6.1 基本操作

- 左クリック: ブロック配置開始
- 左ドラッグ: 通過したマスへ連続してブロック配置
- Q キー: アプリ終了
- W キー: すべての水を強制削除し、10 フレームのクールダウンに入る

6.2 ブロック描画方式

- マウス座標は 8 px 単位でグリッドにスナップする
- 直前マスと現在マスの間を Bresenham の線分補間で埋める
- これによりドラッグが速くても線状にブロックを置ける


7. 更新順序

元コードから読み取れる 1 フレームの更新順は以下の通り。

1. 終了入力を処理する
2. マウスドラッグ状態を更新する
3. 左クリックまたは左ドラッグ中ならブロックを配置する
4. 既存ブロックの寿命を更新する
5. W キー押下時は全水削除してクールダウン開始
6. クールダウン中なら残りフレームを減らす
7. それ以外かつ `frame_count % WATER_SPEED == 0` なら:
   - 排水口周辺の水を除去
   - 水源から新規水を生成
   - 既存水を移動

再実装でもこの順序をベースにしてよいが、排水処理と移動処理の前後関係は明示的に固定すること。


8. 衝突判定仕様

対象セルが以下のいずれかなら「塞がっている」とみなす。

- 固定壁
- 水源
- 排水口
- ブロック
- 水

近傍判定は上下左右 4 方向のみとする。
斜め方向は使用しない。


9. 水の生成・消滅仕様

9.1 生成

- 毎更新タイミングで、各水源から水を 1 個ずつ生成しようとする
- 総水数が `MAX_WATER = 300` 未満のときのみ生成する
- その座標に既に水がある場合は追加生成しない

9.2 排水

- 各排水口について、上下左右に隣接する水を削除する
- 排水口自身のセルに水が入る設計にはしない

9.3 全削除

- W キーで全水を消す
- 直後 10 フレームは再生成しない


10. 描画仕様

10.1 背景タイル

- 固定壁: 茶色の矩形
- 水源: 黄色の三角形
- 排水口: 赤色の三角形
- グリッド線: 緑

10.2 水

- 通常時は青色の矩形
- `status == 1` の場合は赤色だが、元コードでは `status` が変化しないため常に青でよい

10.3 ブロック

- 通常時は灰色
- 寿命低下に応じて白との点滅を行う

10.4 デバッグ表示

- フレーム数
- マウス座標
- ブロック数
- ドラッグ中フラグ
- 水数


11. 元コードから判明した問題点

11.1 仕様上の未完成

- 水が横に流れるとき、塞がれていた場合に逆方向へ切り替える処理が実装途中
- `status`, `stress`, `around`, `birth` など未使用または設計途中の属性が残っている
- ゲームの目的、クリア条件、失敗条件がない

11.2 実装上の危険点

- リストを走査しながら同じリストから削除している
  - `remove_water`
  - `remove_water_all`
  - `Block.update` 内の `block_list.remove(self)`
- これにより要素が飛ばされる可能性がある

- `get_around` は境界チェックをしていない
  - `y-1`, `x-1`, `y+1`, `x+1` を直接参照している
  - 外周が壁で囲まれている前提に依存しており、別ステージでは壊れやすい

- `Block` 生成時の範囲チェックが片側のみ
  - `x < MAX_X`, `y < MAX_Y` は見るが、0 未満を見ていない

- 変数名の typo が多い
  - `mouse_drug` は実質 `mouse_drag`
  - `glid_x`, `glid_y` は `grid_x`, `grid_y`

- 描画順のため、ブロック・水の上に背景が重なる
  - 元コードでは背景が最後に描かれている
  - 実際には背景 -> エンティティ -> UI の順が自然

11.3 設計上の問題

- グローバル変数依存が強い
- ロジックと描画が密結合
- ステージ定義がコード直書き
- テストしにくい


12. 再実装向けの推奨仕様

12.1 設計方針

- 「見た目を持つアプリ層」と「純粋なシミュレーション層」を分離する
- 水・ブロック・マップを 2 次元配列または集合で管理し、検索を O(1) に寄せる
- 1 フレームの処理順を固定し、副作用の場所を限定する
- Pyxel 依存コードは入出力と描画のみに閉じ込める

12.2 推奨モジュール構成

- `config`
  - 定数定義
- `stage`
  - タイル種別
  - ステージロード
  - 境界判定
- `model`
  - `WaterParticle`
  - `TempBlock`
- `simulation`
  - 水生成
  - 排水
  - ブロック寿命更新
  - 水移動解決
- `input`
  - マウスから配置対象セル列を生成
- `renderer`
  - Pyxel 描画
- `app`
  - メインループ統合

12.3 内部データ表現の推奨

- 固定タイル: `stage[y][x]`
- ブロック: `{(x, y): block}`
- 水: `{(x, y): water}` または `set[(x, y)]`

理由:
- 元コードの `exist_list` は毎回線形探索で遅く、AI 実装でもバグを生みやすい
- 座標キー管理なら重複防止と削除が簡単


13. AI 実装しやすい機能要件

13.1 必須要件

- 31x23 グリッドの初期ステージをロードできる
- 水源から水が継続生成される
- 水は下優先、次に左右へ移動する
- ブロックはドラッグで連続配置できる
- ブロックは寿命で消える
- 排水口隣接の水は削除される
- W キーで全水削除と再生成クールダウンがある
- Q キーで終了できる
- デバッグ表示を ON/OFF できる

13.2 推奨要件

- ステージデータを外部ファイル化する
- 乱数シードを固定可能にして再現性を持たせる
- 水の更新順を deterministic にする
- 描画順を背景 -> ブロック -> 水 -> UI に統一する
- 境界外参照が起きないようにする

13.3 非要件

- 物理的に厳密な流体計算
- アニメーション付きタイトル画面
- サウンド
- スコア・クリア判定


14. 水移動アルゴリズムの確定仕様案

AI 実装を安定させるため、元コードの曖昧部分を以下で確定する。

- 各水は 1 更新で最大 1 マスだけ移動する
- 優先順位:
  1. 下
  2. 左右のうち現在の横向き優先方向
  3. 反対側
  4. 停止
- 初めて横移動が必要になったとき、左または右をランダムに選ぶ
- 片側が塞がっていれば反対側を試す
- 両側とも塞がっていれば停止する
- 移動先が同フレーム中に競合した場合:
  - 先着優先でよい
  - ただし更新順は固定すること

この仕様にすると、元コードの「未完成だが意図は見える」部分を自然に補完できる。


15. テスト観点

- 水源の真下が空なら毎フレーム下へ進む
- 水源の真下を塞ぐと左右へ流れる
- 左右片側だけ空いている場合、空いている側へ流れる
- ブロック寿命が 200 フレームで切れる
- ドラッグで斜め線を引いても隙間なくブロックが置かれる
- 排水口の上下左右に来た水が消える
- W キー後 10 フレームは新しい水が出ない
- 境界付近でも IndexError が起きない
- 同一座標に水やブロックが重複生成されない


16. 実装優先順

1. ステージモデルと座標系を確定する
2. 衝突判定 API を純粋関数で作る
3. ブロック配置と寿命更新を実装する
4. 水生成と排水を実装する
5. 水移動ロジックを実装する
6. 描画とデバッグ表示を載せる
7. 入力クールダウンとリセットを整える


17. 最終判断

`original/distillate.py` は「プレイヤーが一時ブロックで水流を誘導する Pyxel 製グリッドシミュレーション」の試作版と判断できる。
再設計時は元コードの行単位再現よりも、以下の 3 点を守る方が重要である。

- グリッド・タイル・エンティティの責務分離
- 安全な更新順序と削除方法
- 曖昧だった横流れ仕様の明文化

この文書を基にすれば、AI に対しては「Pyxel で動く最小実装を作る」「ロジック層を先に実装する」「外部ステージ対応へ拡張する」などの段階的な依頼がしやすくなる。
